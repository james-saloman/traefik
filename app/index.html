Reverse Proxy · Load Balancer · API Gateway

One front door for all your backend services.

Traefik sits at the edge of your infrastructure. Every request from the outside world hits Traefik first, and it decides which backend service should handle it — based on the domain, the path, headers, and more. The killer feature: it auto-discovers your services and configures itself. No restarts, no manual config edits.

It's a single Go binary that runs as a container itself, watching providers like Docker, Kubernetes, or a static config file for changes. The moment you spin up a new container with the right labels, Traefik picks it up and starts routing traffic to it — no reload required. Think of it as the traffic cop standing between your users and every service you run: it terminates TLS, applies routing rules, and forwards the request to a healthy backend.

Why do we even need this?

Imagine you deploy 10 microservices in containers. Each runs on some random internal port. How does the outside world reach them cleanly, securely, and with one certificate?

Without a reverse proxy

  • Clients must know each service's IP + port (app:3001, api:8080…)
  • TLS/HTTPS certificates managed per service
  • No single place for auth, rate-limiting, redirects
  • Scaling or moving a service breaks clients

With Traefik

  • One entry point (e.g. :443) for everything
  • Automatic HTTPS via Let's Encrypt
  • Cross-cutting logic in reusable middlewares
  • Auto-discovers containers via labels — self-configuring

The 4 core building blocks

A request flows left-to-right through these, in order. Learn these four and you understand Traefik. Each card has a small looping animation — watch what glows or moves.

1

Entrypoints

Analogy: the only doors into the building. It doesn't matter how many services live inside — nothing gets in unless it walks through door :80 or door :443.

Entrypoints are the network ports Traefik binds to and listens on — usually web on :80 and websecure on :443. Every request's very first stop, before any routing rule or middleware even looks at it.

:80
:443

Both doors are always open — Traefik listens on both at once.

2

Routers

Analogy: the receptionist with a rulebook. "Asking for shop.local? Floor 3. Asking for api.local? Floor 7."

Routers inspect the incoming request — Host header, path, headers — against rules you define. The first router whose rule matches wins and points to exactly one service. No match means no destination: a 404.

GET
→
Host(`shop.local`)
Host(`api.local`)

Only the matching rule lights up — one router wins per request.

3

Middlewares

Analogy: the checkpoints between the front door and the desk — like airport security. Show your ID (auth), get scanned (rate-limit), get your ticket stamped (headers) — all before you reach the gate.

A router can attach a chain of middlewares that run in order before the request reaches the service: auth, rate-limiting, header injection, path rewriting, and more. Define one once, reuse it on five different routers.

auth
rate
hdrs

One request, one chain — auth, then rate-limit, then headers, in order.

4

Services

Analogy: the department behind the doors — a support team with three identical reps. The receptionist doesn't care which rep you get, just that you get one who's free.

A service is where the request actually ends up — one or more backend containers. With multiple replicas, Traefik load-balances across them (round-robin by default) and skips any replica that fails its health check.

Round-robin: each replica takes its turn getting the next request.
